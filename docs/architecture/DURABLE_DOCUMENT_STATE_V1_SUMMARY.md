# Durable Document State v1 Summary

**Milestone:** v0.12
**Status:** Implemented and verified; closed pending owner commit and tag

## Milestone Purpose

v0.12 adds local, file-backed durability behind the persistence-neutral Document State repository interfaces introduced in v0.11. It proves schema migration, relational mapping, transaction, version, idempotency, reopen, and basic concurrent-writer behavior without changing API, Query Facade, or Streamlit contracts.

SQLite is the verified local/development backend. It is not presented as the production database target.

## Delivered Capabilities

- Persistence contracts and privacy-safe persistence errors.
- Explicit backend configuration for active `in_memory` and `sqlite` modes, with `future_postgres` deferred.
- Deterministic schema metadata for ten Document State record tables plus the migration ledger.
- Immutable migration definitions and applied-migration ledger contracts.
- File-backed standard-library SQLite connection layer with foreign-key mode, bounded busy timeout, WAL mode, and short-lived transactions.
- Explicit relational SQLite schema; complete records are not stored as opaque JSON blobs.
- SQLite mappers that store only canonical validated metadata JSON and reconstruct immutable records through existing contracts.
- Durable read/write repository implementations for all ten record families.
- Ordered, transactional, idempotent migrations with SHA-256 checksum verification and drift rejection.
- Transaction-consistent count/page reads and atomic writes.
- `expected_version` compare-and-swap updates for mutable records.
- Append idempotency using stable keys, canonical content hashes, and database uniqueness constraints.
- Shared in-memory/SQLite repository conformance verification.
- Durability across connection close, repository reconstruction, and migration replay.
- Basic deterministic concurrent-writer verification without sleeps.
- Explicit frozen `DocumentStateComposition` with separate reader and writer surfaces.
- Fail-closed backend selection with no silent SQLite-to-memory fallback.

## Current Architecture

```text
Document Intelligence API
  -> FacadeDocumentIntelligenceProvider
  -> Workflow Query Facade
  -> DocumentStateQueryFacadeAdapter
  -> Document State repositories
  -> DocumentStateComposition
  -> SQLite durable backend OR in-memory backend
```

Composition selects the repository implementation explicitly. It does not activate a backend for API, Streamlit, or workflow writers automatically.

## Runtime Boundaries

- API and Streamlit must not import Document State persistence modules directly.
- Workflow Query Facade must not import persistence modules directly.
- Durable implementations remain behind Document State repository interfaces.
- `DocumentStateComposition` is the explicit backend selection boundary.
- SQLite remains local/dev durability only.
- PostgreSQL and Supabase remain future production options.
- Unknown, deferred, unavailable, or invalid backends fail without silent fallback.

## Privacy And Safety

Durable query-facing tables contain explicit operational summary columns and canonical allowlisted scalar metadata only. They exclude raw document content, raw rows, raw correction values, arbitrary artifacts, stack traces, credentials, and storage internals. Driver failures map to stable safe errors without SQL, values, paths, or trace details.
