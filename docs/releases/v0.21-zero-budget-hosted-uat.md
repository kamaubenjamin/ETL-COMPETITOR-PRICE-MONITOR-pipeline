# v0.21 Zero-Budget Hosted UAT

## Release Status

Phase 6 is complete and Phase 7 release closure is in progress pending owner approval, documentation commit, push, and annotated tag. This release is a **UAT / Technical Preview**, not production.

## Milestone Purpose

Provide a KSh 0, bounded, read-only hosted FlowSync Document Intelligence foundation for synthetic and non-confidential UAT data. The milestone proves deployment, identity, tenant authorization, browser session bootstrap, protected document reads, and safe failure behavior without activating document-processing capabilities.

## Delivered Capabilities

- Separate Vercel deployments for the FlowSync Vite SPA and stateless FastAPI application.
- Supabase UAT email/password identity and minimal tenant/membership foundation.
- Asymmetric JWT/JWKS verification with issuer, audience, signature, and time validation.
- RLS-constrained active tenant membership resolution using the authenticated user context.
- Fixed 10-permission owner catalog composed by the API.
- Authenticated owner session and protected, tenant-scoped, read-only document listing.
- Strict exact-origin CORS for the stable frontend origin.
- Safe browser configuration validation for the Supabase URL, publishable key, API origin, and UAT label.
- Serialized, bounded, generation-aware session bootstrap with controlled cancellation and retry.
- Native browser fetch bound to its global receiver for Firefox compatibility.
- Source, dist, hosted-bundle, and high-signal credential scans.
- Hosted health, docs, authentication, session, permission, and protected-listing smoke coverage.

## Hosted Architecture

| Surface | Stable URL | Responsibility |
| --- | --- | --- |
| FlowSync frontend | `https://flowsync-document-intelligence-uat.vercel.app` | Static SPA, Supabase browser session, safe protected-route state |
| Document Intelligence API | `https://flowsync-document-intelligence-api.vercel.app` | JWT verification, tenant membership, permissions, protected read-only API |
| Supabase Free | Project-specific managed origin | UAT Auth and minimal tenant/membership records behind RLS |

The browser authenticates with the official Supabase client and sends an in-memory bearer token to FastAPI. FastAPI verifies the token, resolves the active membership, derives permissions, and remains the authorization authority. No privileged Supabase key belongs in the browser.

## Security Posture

- Synthetic, non-confidential UAT data only.
- Existing-user sign-in only; no production identity or customer data.
- Asymmetric JWT verification; no shared JWT secret in the frontend.
- Exact-origin CORS with no wildcard tenant or identity authority.
- Tenant and permission decisions remain server-side and fail closed for genuine 401/403 responses.
- No token, password, service-role/secret key, database URL, or complete credential is tracked or embedded in the verified bundle.
- Hosted mutations, uploads, processing, persistence, and external integrations remain disabled.

## Resolved Deployment And Authentication Incidents

- A literal `VITE_SUPABASE_PUBLISHABLE_KEY` name was deployed instead of its configured value; the corrected bundle embeds the validated public value.
- Transient session-profile failures were incorrectly classified as forbidden; genuine unavailable and 403 states are now distinct.
- Duplicate auth events could start overlapping session bootstrap requests; bootstrap is now single-flight and generation-aware.
- Firefox rejected a detached native `fetch` receiver before the API request started; the client binds native fetch to `globalThis`.
- The FastAPI serverless bundle exceeded provider limits through an eager Workflow Runtime-to-pandas/NumPy import chain; the public facade now lazily loads runtime implementations and the minimal API manifest is restored.
- The frontend package manifest and npm lockfile drifted; the committed lockfile is synchronized and clean `npm ci` succeeds.

## Known Limitations

- UAT / Technical Preview only; no production SLA.
- Synthetic and non-confidential data only.
- Read-only hosted foundation; no hosted uploads or write operations.
- No object-storage document workflow or persistent hosted document store.
- No OCR or LLM extraction.
- No asynchronous job, queue, worker, or serverless background-processing runtime.
- Process-local API and Workflow Studio state is ephemeral across recycle or deployment.
- Vercel/Supabase free-tier cold starts, pauses, quotas, log retention, plan terms, and availability apply.
- The minified frontend JavaScript remains larger than Vite's 500 kB advisory threshold.
- The missing favicon produces a cosmetic, non-blocking 404.
- The Node 22 package engine is correct; a Vercel project-setting mismatch/override warning remains operational debt.
- TestClient coverage emits one upstream Starlette deprecation warning recommending `httpx2`.
- Boundary verification retains two unrelated, pre-existing U+FEFF parse warnings.

## Upgrade And Deployment Notes

No application redeployment, Supabase mutation, Vercel setting change, or environment-variable change is required for this documentation-only Phase 7 closure. If a later authorized redeployment is required, retain the separate project roots, Node 22.x, Python 3.12, exact stable origins, current public-variable names, minimal API requirements, and existing fail-closed authentication composition.

Do not add a service-role/secret key to FlowSync, convert the read-only foundation into a write surface, or reuse production/customer data as part of a release deployment.

## Rollback Notes

- The final verified application code is commit `e4b497e`.
- A documentation-only Phase 7 commit can be reverted without changing the hosted application.
- If an application deployment must be rolled back, select the previously verified Vercel deployment for `e4b497e` rather than changing Supabase users, memberships, roles, keys, or schema.
- Preserve the exact frontend origin in API CORS and do not automatically delete or downgrade Supabase identity data during rollback.
