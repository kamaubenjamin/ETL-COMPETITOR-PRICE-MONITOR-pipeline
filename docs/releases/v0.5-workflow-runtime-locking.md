# Release Notes — v0.5-workflow-runtime-locking

**Date**: 2026-06-05  
**Milestone**: v0.5 Runtime Hardening — Workflow Runtime Locking v1  
**Previous Milestone**: v0.4 Matching Runtime  

---

## Summary

Workflow Runtime Locking v1 addresses the highest-priority known technical debt: preventing duplicate concurrent execution of the same workflow. The Workflow Runtime previously had no mechanism to coordinate overlapping `run()` calls for the same `workflow_id`, creating risks of duplicate extraction, duplicate alerts, duplicate reports, and incorrect aggregated state.

## What's New

### Locking Infrastructure
- **`LockProvider` ABC** with three implementations:
  - `DBLockProvider` — database-backed row-level locking with execution leases (primary, production)
  - `FileLockProvider` — file-based advisory locking with stale detection (fallback, single-host)
  - `MemoryLockProvider` — thread-safe in-memory locking (development, testing)
- **`LockProviderRegistry`** — priority-ordered provider chain with automatic fallback (DB → File → Memory → error)
- **`WorkflowExecutionGuard`** — wraps the `run()` lifecycle with lock acquisition, periodic lease refresh, and lock release
- **`WorkflowIdempotencyRegistry`** — `DBIdempotencyRegistry` and `MemoryIdempotencyRegistry` implementations for deterministic deduplication of completed runs

### Workflow Integration
- **`ExecutionContext`** updated with `lock_acquisition: Optional[LockAcquisition]` and `idempotency_key: Optional[str]` fields
- **`WorkflowResult`** updated with `idempotency_key: Optional[str]` and `lock_status: Optional[str]` fields
- **`WorkflowRunner.run()`** — full lock lifecycle integration when guard is configured (idempotency check → lock → execute → refresh → release → record)
- **`generate_idempotency_key()`** helper function for deterministic key generation from workflow_id and schedule slot

### Database Schema
- **Migration 006**: `workflow_locks` table — lock_id (PK), holder_id, acquired_at, expires_at, lease metadata, indices
- **Migration 007**: `workflow_idempotency` table — idempotency_key (PK), pipeline_run_id, status, created_at, completed_at, indices

### Lock Status Values
| Value | Meaning |
|-------|---------|
| `"acquired"` | Lock obtained, execution proceeded |
| `"rejected_busy"` | Another execution holds the lock |
| `"rejected_duplicate"` | Idempotency key already completed |
| `"not_locked"` | No guard configured (original behaviour) |

## Known Issues

1. **Windows file lock timing**: `TestFileLockProvider::test_refresh_before_expiry` fails intermittently on Windows due to sub-millisecond timestamp precision. Affects file lock provider only; DB and memory providers are unaffected.
2. **Registry health-check**: `LockProviderRegistry.resolve(None)` performs a health-check on all providers. For production, call `resolve("database")` for direct access.

## Migration

Run migrations in order before deploying locking code:
```bash
# Apply migrations 006 and 007
python -c "
import sqlite3
conn = sqlite3.connect('your_history_store.db')
conn.executescript(open('scripts/migrations/006_create_workflow_locks_table.sql').read())
conn.executescript(open('scripts/migrations/007_create_workflow_idempotency_table.sql').read())
conn.commit()
"
```

Locking is opt-in. Default `LOCK_PROVIDER=memory` means no behaviour change until explicitly configured.

## Rollback

```bash
# Disable locking without code change:
# Set LOCK_PROVIDER=memory in environment and restart workers

# Permanent rollback:
# 1. Revert WorkflowRunner.run() to pre-locking lifecycle
# 2. Revert ExecutionContext and WorkflowResult contracts
# 3. Drop workflow_locks and workflow_idempotency tables
```
