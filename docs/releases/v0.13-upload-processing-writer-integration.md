# v0.13 Upload-to-Processing Writer Integration

## Release Name

Upload-to-Processing Writer Integration v1

## Capabilities Added

- Runtime-neutral Document State writer package with immutable commands.
- Stable privacy-safe writer result and error contracts.
- Deterministic domain-separated idempotency helpers.
- Governed mapping catalog and bounded opaque artifact references.
- Explicit internal writer ports with repository injection.
- Ingestion writer for document creation, lifecycle, classification snapshots, and audit.
- Processing writer for processing snapshots, validation issues, and matching summaries.
- Review writer for review, correction, and reprocess summaries.
- Workflow writer for workflow runs, lifecycle events, and audits.
- Replay-safe append behavior, explicit expected versions, and bounded partial-retry outcomes.
- In-memory and SQLite backend parity.
- Read-after-write verification through Query Facade and the existing API provider.

## Architecture

```text
Producer / internal runtime output
  -> Document State writer services
  -> Document State repositories
  -> DocumentStateComposition
  -> SQLite durable backend OR in-memory backend
  -> DocumentStateQueryFacadeAdapter
  -> Workflow Query Facade
  -> Document Intelligence API
  -> Streamlit api_preview
```

Concrete runtime producer adapters remain deferred. Writer services do not select repository backends.

## Compatibility Notes

This release is additive. v0.9 API paths, GET-only methods, payload meanings, envelopes, pagination, request IDs, and security headers remain unchanged. Streamlit `local_preview` and `api_preview` behavior remains unchanged. No public mutation route, API write path, or UI mutation was introduced.

## Privacy And Retry Notes

Writer records exclude raw documents, rows, correction values, artifact payloads, storage paths, credentials, stack traces, and raw exception messages. Append records use deterministic keys; mutable snapshots use explicit expected versions; verified partial retries resume safely without duplicate records.

## Known Limitation

Lifecycle events are append-only and do not yet advance the mutable document projection, which remains `received`. A governed lifecycle-to-snapshot transition policy is deferred.

## Known Non-Goals

- Concrete runtime producer adapters and production bootstrap activation.
- Public mutation endpoints or upload UI/API.
- Cross-record unit of work or transactional outbox.
- Auth/authz, tenant isolation, or production composition.
- PostgreSQL/Supabase or raw encrypted blob storage.
- FlowSync Document Intelligence UI.
- OCR, LLM processing, or external services.
