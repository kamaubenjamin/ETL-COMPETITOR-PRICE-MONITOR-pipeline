# v0.14 Lifecycle Snapshot Advancement

## Release Name

Lifecycle Snapshot Advancement v1

## Capabilities Added

- Immutable lifecycle transition, decision, recovery-policy, result, and error contracts.
- Explicit deterministic transition catalog over existing document statuses.
- Repository-neutral `LifecycleAdvancementService` with optimistic projection updates.
- Governed status, current-stage, update-time, and version advancement.
- Same-status replay no-op and terminal `exported` handling.
- Explicit failed-state recovery policy requirement.
- Optional lifecycle-service injection across ingestion, processing, review, and workflow writers.
- Audit-first append-then-advance behavior.
- Privacy-safe `projection_pending` result after a committed event and projection conflict.
- Replay-based projection repair without duplicate lifecycle history.
- In-memory and SQLite backend equivalence and SQLite reconstruction verification.
- Read-after-advance verification through Query Facade and v0.9-compatible API-provider shapes.

## Architecture

```text
Writer services
  -> LifecycleAdvancementService
  -> DocumentRecord projection update
  -> Document State repositories
  -> DocumentStateQueryFacadeAdapter
  -> Workflow Query Facade
  -> Document Intelligence API
  -> Streamlit api_preview
```

Append-only lifecycle events remain the audit truth. `DocumentRecord` is the mutable current projection. Writers receive services and repositories explicitly and do not select backends.

## Compatibility Notes

This release is additive. v0.9 API paths, GET-only methods, payload meanings, envelopes, pagination, request IDs, and security headers remain unchanged. Streamlit `local_preview` and `api_preview` behavior remains unchanged. Existing writers preserve append-only behavior when no lifecycle service is injected. Review records remain independently governed after document approval or export.

## Privacy And Safety

Lifecycle and read projections exclude raw documents, rows, correction values, artifact payloads, storage paths, credentials, stack traces, and raw exception messages. Metadata remains bounded and scalar-only. Projection conflicts expose stable codes rather than repository or driver details.

## Known Non-Goals

- Runtime producer adapters and production composition activation.
- Repository-backed recovery-plan verification.
- Cross-record transaction, outbox, or background reconciliation.
- Public mutation endpoints or upload UI/API.
- Authentication, authorization, or tenant isolation.
- PostgreSQL/Supabase production activation.
- Encrypted raw blob storage.
- FlowSync Document Intelligence.
- OCR, LLM processing, or external services.
